# REQUIREMENTS.md

## Overview
**Project:** code‑sentinel  
**Purpose:** Provide an automated tool that detects, classifies, and mitigates adversarial AI‑generated code within software repositories. The system must integrate seamlessly into typical CI/CD pipelines, support multiple programming languages, and enforce configurable security policies to protect downstream builds and production environments.

---

## 1. Functional Requirements  

| ID | Description | Acceptance Criteria |
|----|-------------|----------------------|
| **FR‑1** | **Code Ingestion** – Accept source code from supported entry points. | • Supports Git checkout, tarball upload, and direct file‑system paths.<br>• Detects language automatically (Python, JavaScript/TypeScript, Go, Rust, Java, C/C++). |
| **FR‑2** | **Adversarial Code Detection** – Identify code that is likely generated by malicious LLM prompts. | • Uses a pre‑trained classifier (based on the `instr‑resp` and `auto` datasets) with ≥ 90 % precision on a held‑out validation set.<br>• Returns a confidence score (0‑100) for each file/snippet. |
| **FR‑3** | **Risk Scoring & Policy Engine** – Map detection confidence to a risk level and enforce policies. | • Configurable thresholds: `low`, `medium`, `high`.<br>• Policy actions: `log`, `fail‑build`, `auto‑sanitize`, `quarantine`. |
| **FR‑4** | **Mitigation Actions** – Provide automated remediation options. | • **Auto‑sanitize**: Replace suspicious constructs with safe equivalents (e.g., remove `eval`, network calls, unsafe system calls).<br>• **Quarantine**: Move flagged files to a separate branch/ directory for manual review.<br>• **Report Generation**: Produce a SARIF‑compatible report summarizing findings. |
| **FR‑5** | **CI/CD Integration** – Offer plugins for major pipelines. | • GitHub Actions, GitLab CI, Azure Pipelines, Jenkins.<br>• Exit code semantics: `0` = clean, `1` = warnings, `2` = failure (policy‑enforced). |
| **FR‑6** | **Developer Feedback Loop** – Surface actionable insights to developers. | • Inline comments on pull‑requests indicating line‑level issues.<br>• CLI `--explain` flag that prints model rationale for each detection. |
| **FR‑7** | **Model Update Mechanism** – Allow periodic refresh of detection models without downtime. | • Pull new model artifacts from an internal model registry (e.g., S3 bucket).<br>• Validate checksum before activation. |
| **FR‑8** | **Audit Trail** – Record all detection events for compliance. | • Immutable log (JSONL) stored in a configurable backend (local file, Elasticsearch, or cloud log service).<br>• Include timestamp, commit SHA, file path, confidence, action taken. |
| **FR‑9** | **Configuration Management** – Centralized, version‑controlled config. | • YAML config file (`code-sentinel.yml`) supporting per‑repo overrides.<br>• Schema validation on load. |
| **FR‑10** | **Multi‑language Extensibility** – Enable adding new language parsers. | • Plugin interface (`LanguageAdapter`) with methods `detect_language`, `extract_ast`, `sanitize`.<br>• Documentation and example adapter for a new language. |

## 2. Non‑Functional Requirements  

| ID | Category | Requirement |
|----|----------|-------------|
| **NFR‑1** | **Performance** | • End‑to‑end analysis of a typical 10 k LOC repository ≤ 30 seconds on a single vCPU (baseline).<br>• Parallel processing across files when `--workers N` is specified. |
| **NFR‑2** | **Scalability** | • Horizontal scaling via Docker Swarm/Kubernetes – stateless workers can be added to handle > 10 k concurrent analyses. |
| **NFR‑3** | **Security** | • All external dependencies must be vetted (license‑compatible, no known CVEs).<br>• Execution sandboxed using `gVisor` or similar to prevent malicious code execution during analysis.<br>• Configurable secret redaction – any detected hard‑coded credentials are masked in logs/reports. |
| **NFR‑4** | **Reliability** | • 99.9 % availability of the CLI/service when deployed in production.<br>• Automatic retry (up to 3 attempts) for transient network failures when pulling models or uploading reports. |
| **NFR‑5** | **Observability** | • Emit Prometheus metrics: `files_processed`, `detections_total`, `high_risk_detections`, `analysis_latency_seconds`.<br>• Structured logs in JSON format. |
| **NFR‑6** | **Maintainability** | • Code coverage ≥ 85 % (unit + integration).<br>• Follow Pylint/ESLint style guides; CI enforces linting. |
| **NFR‑7** | **Portability** | • Distributable as a single‑file executable via `pyinstaller` (Linux/macOS/Windows) and as a Docker image (`docker.io/axentx/code-sentinel:<tag>`). |
| **NFR‑8** | **Compliance** | • All data processing must comply with GDPR – no personal data persisted beyond the analysis run. |
| **NFR‑9** | **Usability** | • CLI must provide `--help` with examples.<br>• Exit codes documented; error messages actionable. |
| **NFR‑10** | **Extensibility** | • Plugin system for custom detection models (e.g., proprietary LLM fine‑tuned on internal code). |

---

## 3. Constraints  

1. **Runtime Environment** – Must run on Linux (kernel ≥ 5.4) and be compatible with the corporate CI runners used by Axentx (Ubuntu‑22.04 LTS).  
2. **Dependency Licenses** – All third‑party libraries must be Apache‑2.0, MIT, BSD, or similarly permissive; no GPL‑v3 unless isolated behind a separate optional plugin.  
3. **Model Size** – The default detection model must fit within 2 GB RAM to allow deployment on modest CI workers.  
4. **Data Privacy** – Source code must never be uploaded to external services; all analysis is performed locally or within the customer’s trusted network.  
5. **Release Cadence** – Align with Axentx’s quarterly release schedule; each major version must be backward compatible with config files from the previous major version.  

---

## 4. Assumptions  

| ID | Assumption |
|----|------------|
| **A‑1** | Repositories contain a `.git` directory or are otherwise version‑controlled, enabling commit SHA extraction for audit logs. |
| **A‑2** | Developers have the ability to add a `code-sentinel.yml` file at the repository root. |
| **A‑3** | The underlying LLM‑based classifier can be refreshed quarterly without breaking the API contract. |
| **A‑4** | CI runners provide at least 2 CPU cores and 4 GB RAM for the analysis step. |
| **A‑5** | Security teams will approve the sandboxing approach (gVisor or similar) before production rollout. |
| **A‑6** | Existing Axentx logging infrastructure can ingest JSONL audit logs via a configurable endpoint. |
| **A‑7** | The SARIF format is sufficient for downstream security tooling (e.g., GitHub Code Scanning). |

---

## 5. Acceptance Criteria Summary  

- All functional requirements FR‑1 – FR‑10 are implemented and demonstrated in an end‑to‑end CI pipeline run on a sample multi‑language repository.  
- Non‑functional targets NFR‑1 – NFR‑10 are met as measured by automated performance tests, security scans, and CI quality gates.  
- Documentation includes a quick‑start guide, configuration reference, and plugin development tutorial.  
- The product passes internal security review and is packaged as both a Docker image and a single‑binary executable.
